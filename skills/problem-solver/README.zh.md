# 问题诊断师 (Problem Solver)

**Language / Ngôn ngữ / 语言:** [English](./README.md) | [Tiếng Việt](./README.vi.md) | [中文](./README.zh.md)

一个陪你的 AI 助手一起做诊断的伙伴——在帮你解决任何问题之前，它先帮你搞清楚到底哪里出了问题，以及为什么。可以把它想象成夹在"出问题了"和"该怎么办"之间的那道纪律：把一句模糊的抱怨变成一个精确、经过确认的问题陈述，追踪真正的原因而不是第一个听起来合理的说法，然后才交给你去解决它。

## 为什么需要它

问一个普通 AI"为什么销售额下降了？"，它会在一口气之内给你一个听起来很自信的解释——有条理、说得通，而且往往完全是编的，因为 AI 根本没有权限看到你的数据、你的团队、你的客户。一条用编造出来的事实搭建起来的整洁因果链，比一句诚实的"还不知道"更危险，因为它会让你满怀信心地去修错地方。

这个 Skill 反其道而行之：**事实只能来自你，假设必须被标注，在现实世界确认之前，任何东西都不能被称为"原因"。** 会话中的每一条因果论断都会带上 `[verified]`（已验证）或 `[assumed]`（假设）标签，每一条支撑诊断的关键假设都会被记入日志，并附上最便宜的核实方式，并且至少两个对立的假说会一直保留在桌面上，直到证据——而不是投票，也不是直觉——分出胜负。如果你在原因还没查清楚之前就要解决方案，它会明说这一点，并把选择权交给你，而不是悄悄跳过困难的那一步。

## 你可以让它做什么

**有什么地方出问题了，你不知道为什么** —— 最核心的场景。
> "上个月销售额掉了 15%，我们不知道为什么。"
> "订单一直延迟送达。试了几个办法都没用。"

**问题反复出现，或者每次的解决方案都失败了。**
> "这个 bug 修了多少次都还会再出现。"
> "每次修好 onboarding 的流失率，别的地方又出问题。"

**你想要一次有结构的根因排查。**
> "帮我对这次故障做一次 five whys。"
> "陪我做一次事后复盘——昨天生产环境宕机了 2 小时。"

**你直接要解决方案，但原因还没查清楚。**
> "给我一些挽救下滑销售额的点子。" *（skill 会指出这其实是一个披着头脑风暴外衣的诊断请求，并让你自己选方向）*

无论你用中文、越南语、英语，还是任何其他语言，它的工作方式都一样：
> "Why do orders keep arriving late and we don't know why?"
> "Tìm giúp tôi nguyên nhân gốc của việc doanh số giảm."

你不需要说出某个具体方法的名字——"我们不知道为什么 X 一直发生"就足够开始了。

## 一次会话是什么样子

**Frame（定框）。** 一次简洁的了解情况，不是一张表格：用可观察的方式描述发生了什么，从什么时候开始，谁/什么受到影响、代价是什么，已经尝试过什么，"解决了"应该是什么样子。如果你的开场消息已经带来了一个很有价值的线索——一个边界对比加上一个同期发生的变化——它会在第一次回复里就针对这条线索提出一个便宜的核实方法，而不是留到以后。然后它会把精炼后的问题陈述读给你听，剥离掉里面藏着的任何"猜测的原因"或"跳过去的解决方案"，等你确认后再继续。

**Bound（划界）。** 一次 Is/Is-Not 分析：问题出现在哪里 vs. 不出现在哪里，什么时候 vs. 什么时候不，谁 vs. 谁不受影响。最锐利的线索就藏在两栏之间的差异里。

**Diagnose（诊断）。** 它会根据问题的形状选择合适的方法——单一症状、背后可能是一条线性因果链的用 Five Whys；有很多可能因素、没有明显链条的用鱼骨图；反复发作或者越修越糟的用因果回路——然后和你一起、一次一个问题地搭建因果树，每条链接一落地就标上 `[verified]` 或 `[assumed]`。至少两个对立的候选原因会一直留在桌面上，直到某个测试把它们区分开。在领先的候选原因被验证，或者你明确选择带着风险继续之前，不会转向解决方案。

**Solve（求解）。** 一旦原因被确认，它会针对*那个原因*生成解决方案——如果 brainstorm-coach 已安装，会把这个已验证的根因交给它做真正的发散式构思。每个方案都会说明自己针对的是哪个根因；做不到这一点的方案会被标注为"症状补丁"，而不是真正的解决方案。

**Decide（决策）。** 由你来定标准（有效性、成本、时间、风险——你的标准可能不一样）。每个打分都必须引用证据，否则就老老实实标成"未知——假设"。给出一个推荐方案，并说明什么情况会改变这个推荐。

**Plan（规划）。** 一个上线方式（置信度不高时默认走试点），第一步的负责人，与 Frame 阶段"解决了"标准挂钩的成功指标，以及"pivot trigger"（转向触发条件）——一条清晰的规则，说明什么情况下计划不起作用意味着诊断错了，而不只是执行力度不够。

## 证据纪律

这是让这个 Skill 值得信赖、而不只是说得头头是道的关键部分。三条规则贯穿每个阶段：

- 除非是你说出来或展示出来的，否则不会有任何关于你的世界的"事实"被直接写出来。缺失的事实会变成一个问题，或者一个 `[assumed]` 标签——绝不会是一个被悄悄填上的猜测。
- 每个根因候选都带着一个测试：「如果这就是原因，我们应该会看到 ___」，加上验证它是否错误的最便宜方式。一个没有任何观察能够反驳的原因（比如"沟通不畅"）还算不上一个诊断。
- 置信度*下降*也算作一种进步。一个被检验并排除掉的假设，刚刚救了你一把，让你不至于在流沙上建方案——这个 skill 会把它当作一次胜利明说出来，而不是一次挫败。

## 一个具体例子

> **你：** 订单一直延迟送达，大概四分之一吧。从六月初开始的。已经提醒员工手脚快点了，没用。
>
> **Agent：** 在我确认问题陈述之前先快速问一句——延迟是集中在某类订单上，还是分布得比较均匀？还有，从六月初到现在，还有什么变化吗——菜单、人手、快递方？
>
> **你：** 你这么一说我想起来了，主要是定制蛋糕订单。而且六月初确实有一篇菜单帖子火了。
>
> **Agent：** 这个信息很有用——这是我想请你确认的陈述：*"自六月初起，大约四分之一的订单延迟送达，集中在定制蛋糕订单上；同一周有一篇菜单帖子上了热门。"* 对吗？如果没错，在继续之前先做一个便宜的核实：如果定制蛋糕的量增长超过了备货产能，那每个定制蛋糕的平均准备时间应该会逐渐逼近甚至超过配送时间窗口——方便给接下来 5 个订单计个时吗？
>
> **你：** 没错，我去计时接下来 5 个。
>
> **Agent：** *（记下这个计划，把"备货产能 vs. 定制蛋糕订单量"标为领先的 `[assumed]` 候选原因并附上待完成的测试，同时把"快递方服务下滑"作为对立假说继续保留，在这里暂停，等你带着计时结果回来。）*

## 方法库，按问题的形状匹配

`references/diagnosis.md` 收录了方法库——Five Whys、鱼骨图、因果回路与系统原型，外加针对停滞不前的变革项目的 force-field 和约束分析——每种方法都说明了适用场景并配有实例。Agent 会根据你的问题真实的形状来选择（单一症状 vs. 多因素 vs. 一个会"反抗"每次修复的问题），而不是凭习惯，也不是为了显得很全面。

## 与同系列的其他 Skill 配合使用

这个 Skill 属于一个小型 Skill 系列（github.com/tronghieu/agent-skills），这些 Skill 之间会在合适的时机彼此交棒。如果下一个 Skill 还没安装，agent 只会提一下它的名字，告诉你可以从同一个仓库安装，然后继续用一个更轻量的版本顶上。

| 什么时候 | 会交给 |
|---|---|
| 原因已经验证，你需要针对它做真正的发散式构思 | **brainstorm-coach** |
| "问题"其实是关于你的用户是谁、他们需要什么 | **design-thinking** |
| 诊断或决策文档需要一次论证审查 | **critical-thinking** |
| 这个修复方案已经长成了公司级别的战略赌注 | **strategy-board** |
| 某个原因或方案取决于你还没有的市场事实 | **market-researcher** |
| 计划正在变成一个需要跟踪的、真正的多周项目 | **project-manager** |

## 开始使用

把这行命令粘贴到终端里，完成安装：

```bash
npx skills add tronghieu/agent-skills --skill problem-solver
```

然后直接告诉你的 AI agent 哪里出了问题：

```text
上个月销售额掉了 15%，我们不知道为什么。
```

```text
这事儿一直反复发生，什么办法都试过了——帮我找到真正的原因。
```

```text
陪我对昨天的故障做一次事后复盘。
```

```text
Doanh số giảm 15% tháng trước và chúng tôi không biết vì sao.
```

你完全不需要专门说"problem solver"——"我们不知道为什么 X 一直发生"就足够让它接上话头了。

# Project Manager Skill

**Language / Ngôn ngữ / 语言:** [English](./README.md) | [Tiếng Việt](./README.vi.md) | [中文](./README.zh.md)

这个 skill 把你的 AI agent 变成一位有纪律的**项目管理搭档（PM/PMO）**——不会在三个里程碑正悄悄延期的时候,还递给你一份全线飘绿的状态报告,也不会拿一份全凭乐观情绪拼出来的时间线来交差。

## Project Manager Skill 是什么？

让一个普通的聊天机器人帮你"做个项目计划",它几秒钟就能生成一条看起来很自信的时间线——里程碑、日期、依赖关系全部填好,但没有一处是真的。危机当中问它"这周的进度怎么样",它也会欣然把总结涂成绿色,因为状态报告"看起来"就该是绿的。有三种失败模式杀死的真实项目比任何其他原因都多,而一个过于积极的 AI 助手会在默认情况下把这三者都变得更糟:**planning fallacy**(把乐观的估算当成承诺来呈现)、**watermelon reporting**(外面是绿的,里面是烂的——一个追溯不到任何人证据的状态颜色)、以及 **scope creep**(一个个"小"改动被悄悄塞进去,直到 baseline 只剩虚构)。这个 skill 的设计目标,就是让这三种情况都无法悄悄发生:每一个状态颜色、每一个完成百分比、每一个估算,都带着一个可以点开追溯来源的标签——一条记录在案的事实、一个写明区间的假设,或者你本人给出且带日期的说法。没有证据的时候,skill 会问你,或者登记一个诚实的假设,绝不会为了让表格看起来齐整就把某个里程碑随手涂绿。

它也不是一个一次性文档生成器。项目不会在计划交付那一刻就结束——日程要守住,风险要盯着,状态报告要一周接一周地说真话,scope 要守得住,会议要变成有人跟进的行动项,复盘要在同样的错误重演之前跑起来。这个 skill 是作为**一个仍在推进的项目的搭档**运行的:一个工作区,只需初始化一次,你会在接下来的几周、几个月里反复回来使用它。它记得之前决定了什么、为什么这么决定,所以每次开始都是"目前情况是这样",而不是让你从头再解释一遍项目背景。

项目经理始终是你。日程是你的,你在和 sponsor 对话,scope 和日期上的每一个决定都由你拍板。这个 skill 的工作是把草稿写好,在你的要求和已有的登记记录相矛盾时说出来,老老实实做估算和风险计算,并且把账本记清楚,不让任何事情被悄悄遗忘、覆盖,或者反复翻案。

## 为什么要用它？

- **每个状态和数字都交代来源。** 每一个 RAG 颜色、完成百分比、预测日期、估算,都带着标签:`EV#`(记录在案的事实)、`A#`(登记过的假设,带诚实区间),或者 `(user, <日期>)`。没有标签的状态颜色被当作 bug,而不是一份摘要。
- **估算在交付前必须先对照历史。** 任何估算都是一个区间,而不是一个死数字,并且要先对照项目自己的 lesson 记录和过往的 plan-vs-actual 数据——一个无视最近三次延期记录的估算,不是估算,是一厢情愿。
- **scope 的每一次改动都必须过关,没有例外。** 任何涉及 scope、日期或预算的改动,在被接受*之前*都要有一份影响评估和一个被点名的决策人,旧的 baseline 永远保留,绝不覆盖。"先塞进去再说"没有机会悄悄发生。
- **交给你之前会有人复查一遍。** 基线计划、状态报告、变更决策各自都要经过一次带批判性的审查,然后才会请你据此行动。审查发现的问题会写进文档本身,不会被悄悄改掉。
- **工作区替你记住一切,你不必自己记。** 一份只增不改的决策日志和一份状态文件,就是项目的组织记忆:决定了什么、为什么、还有什么在等谁处理。三个月后回来,skill 会先读完这些,再开口说话。

## 五条这个 skill 绝不妥协的原则

1. **每个状态和数字都交代来源。** 每一个 RAG 颜色、完成百分比、预测日期、估算都带着 `EV#`、`A#` 或 `(user, <日期>)` 标签——绝不是为了让表格看起来齐整而随手选的颜色。
2. **估算在交付前必须先对照历史。** 任何估算都是一个区间,写明依据,并且对照项目自己的 lesson 记录或一个明确的 reference class 做检验。缓冲时间是单独可见的一行,绝不是藏在任务里的隐性填塞。
3. **scope 的改动必须过关。** 任何涉及 scope、日期或预算的改动,在被接受*之前*都要有影响评估,以及被点名的决策权威给出的明确决定。重新设定 baseline 是一次被记录的事件;旧的 baseline 被保留,不被覆盖。
4. **工作区是事实的来源,不是记忆。** 每次重新进入项目,skill 都会先读状态文件、最新的状态快照、和近期的决策记录,再做任何事,并用项目自己的语言——真实的里程碑名称、真实的人名——把现状复述给你听。
5. **不经复查,不上线。** 基线计划、状态报告、变更决策在交到你手上之前都要经过一次带批判性的审查——不论审查发现什么,都会展示给你看,不会被悄悄清理掉。

每一份产出物都要经受住这个问题的考验:**"如果 sponsor 去核实这份文档里的每一个数字,哪些站不住脚——决定会因此改变吗？"**

## 一套镜头组成的团队

现在每顶帽子都有了名字,但你从来不需要"召唤"谁——始终是同一位 PM 根据工作需要切换帽子,不是一份供你挑选的人设名单。这些名字取自古希腊哲学家,各自对应他们最广为人知的特质。

| 镜头 | 关注点 | 你会在这些时候遇到它 |
|------|---------|------------------------|
| **Plato**（柏拉图）· Anchor | 方向——守住项目的理想形态:charter、方法论是否合适、"完成"到底意味着什么 | 你要启动一个项目,或者需要在 predictive/agile/hybrid 之间做选择 |
| **Aristotle**（亚里士多德）· Frame | 结构——WBS/backlog、日程、依赖关系、关键路径 | 你要一份项目计划、时间线,或 WBS |
| **Pythagoras**（毕达哥拉斯）· Gauge | 估算——只信有来源可查的数字 | 你问某件事要花多久 |
| **Epictetus**（爱比克泰德）· Watch | 风险——分清哪些是你能控制的,哪些是你要接受的（ROAM） | 你要求建立或复查风险登记表 |
| **Diogenes**（第欧根尼）· Pulse | 说真话——提着灯笼去找那些外绿内烂的报告 | 你要一份状态报告 |
| **Zeno**（芝诺）· Gate | 变更控制——不放过任何未经审视就想通过的改动 | scope、日期或预算想要变动 |
| **Socrates**（苏格拉底）· Bridge | 人——利益相关者地图、沟通计划、把会议变成可跟进的行动、靠对话推进工作 | 你需要一份利益相关者地图,或想把一场会议变成行动项 |
| **Heraclitus**（赫拉克利特）· Loop | 学习——深知计划就像一条河;复盘反哺下一次估算 | 你要求做一次复盘,或记录 lesson |
| **Solon**（梭伦）· Judge | 审计——立下所有产出物都必须遵守的法则 | 自动触发,在一份计划、状态报告或变更决策交付之前 |

只要 agent 能拉起一个独立的 subagent,Judge（Solon）就会作为独立的一轮跑起来——自己批改自己的作业没多大意义,所以它刻意不给自己打分。其余的一切,都是同一位 PM 在对话中和你一起推进,全程都在场。

## 你可以提哪些要求

没有固定顺序,也没有"五个阶段走第一步"这种说法——你从当下需要的任何一点切入,skill 会自动路由到合适的镜头:

- **"启动这个项目"** —— Anchor 镜头:一份 charter,包含目标、scope 的取舍、sponsor 和决策权威,如果你还没定方法论,也会帮你一起选。
- **"做一份项目计划 / 时间线 / WBS"** —— Frame 镜头:一份贴合你所选方法论形态的日程——WBS 加关键路径、或 backlog 加 release、或一条里程碑主线配上内部的迭代循环。
- **"估算这件事要多久"** —— Gauge 镜头:一个带明确依据的区间估算,对照项目自己的 lesson 或一个 reference class 做检验,缓冲时间作为单独可见的一行。
- **"建一份风险登记表"** —— Watch 镜头:概率 × 影响、ROAM 分类、每条风险都有点名的负责人,风险够高时还会做一次 pre-mortem。
- **"写这周的状态报告"** —— Pulse 镜头:一份每个颜色都引用证据的 RAG 报告,坏消息排在最前面,预测日期对照延期历史来定。
- **"deadline 延期了,现在怎么办"** —— Gate 镜头:一份覆盖日程/成本/风险/质量的影响评估,提交给被点名的决策权威,无论结果如何 baseline 都会被保留。
- **"准备 steering committee 会议"** / **"跟进这场会议的行动项"** —— Bridge 镜头:一份按利益相关者分层的沟通计划,或者把会议纪要变成有人负责、有截止日期的行动项。
- **"做一次复盘"** / **"记录这个 sprint 的 lesson"** —— Loop 镜头:一条明确写出它该改变哪个估算类别或检查清单的 lesson,而不是一条流水账。
- **"审视我的项目 portfolio"** —— PMO 扩展模块:对你指定的每个项目做一次汇总,颜色按最可信证据(而非平均值)来定,并把资源冲突和阶段性关卡建议摆出来,交给你决定。

一次性的请求也会被认真对待、不需要繁文缛节——你不必为了得到一份干净的状态报告就先搭建整个工作区——但即使是快速处理,数字依然会被贴上标签。

## 它会生成的工作区

```bash
bash scripts/init-project.sh "<项目名称>" [父目录]
```

这个脚本是幂等的——在一个已经用了好几个月的工作区上重新运行也很安全,因为它绝不会覆盖已存在的文件。它总是在一个固定路径下创建工作区:`<父目录>/_project/`。

```text
_project/
  context/                # 变化缓慢的事实,随各个 play 的需要逐步填充
    charter.md            # 项目为何存在:目标、scope 的取舍、成功标准、sponsor
    methodology.md        # predictive / agile / hybrid ——选择 + 定制 + 节奏
    team.md                # 人员名单、角色、产能、RACI
    stakeholders.md        # 利益相关者地图 + 沟通计划
    environment.md         # 工具、日历、外部依赖、约束条件
    glossary.md            # 项目自己的词汇表
  registers/              # 持续更新的登记簿——每周都在变
    risks.md               # R# 概率 × 影响、ROAM、负责人、触发条件
    decisions.md            # 只增不改的决策日志——回归项目时的支柱
    changes.md              # CH# scope/日期/预算的改动,附影响评估 + 决策
    actions.md              # AC# 行动项:负责人、截止日期、来源、状态
    assumptions.md          # A# 内容、依据、区间、用在哪里、状态
    lessons.md              # L# plan-vs-actual 的 lesson,反哺未来的估算
    evidence.md             # EV# 事实登记:demo 通过、导出文件收到、达成某个日期
  plan/
    schedule.md            # M# 里程碑、依赖关系、关键路径
    estimates.md            # 带依据标签和 reference class 的区间估算
    baseline-<date>.md      # 冻结的快照——每次设定或重设 baseline 时都会保留
  status/
    status-<date>.md        # 按日期归档的 RAG-带证据快照——保留,从不覆盖
  state.md                  # 目前打开的 play、正在等你处理的事项、上一次会话
```

**回来接着做很轻松。** 走进一个已经存在的 `_project/`,提出任何问题——skill 会先读 `state.md`、最新的状态快照、以及近期的决策记录,在做任何事之前先把现状说给你听:"里程碑 M3 因为 vendor API 延期(R4,由 Priya 负责)被标记为黄色;有一个变更请求正在等 sponsor 拍板;有三个行动项已经过期。"它更相信文件,而不是自己的记忆,也不会悄悄重做或覆盖你已经完成的工作。

## 方法论是背景,不是身份

这个 skill 本身是通用的;*你的*项目才有方法论。`initiate` 会记录下这个选择——predictive、agile 还是 hybrid——之后每个 play 都会据此调整自己的产出形态。一个 skill,三种方言:

| Play | Predictive | Agile | Hybrid |
|------|-----------|-------|--------|
| plan | WBS、Gantt 风格日程、关键路径 | backlog、release/sprint、产能 | 里程碑主线 + 内部迭代循环 |
| estimate | 自底向上 + reference class | 基于 velocity + reference class | 两者兼顾,互相印证 |
| report-status | 里程碑/EVM 风格 | burnup、流程指标 | 里程碑主线 + 流程细节 |

无论哪种方言,这些原则都不会改变:一份用编造的 velocity 画出来的 agile burnup 图,和一份用编造日期画出来的 Gantt 图一样虚假。

## 与伙伴 skill 协同工作

项目交付天天都会碰到产品决策和公司级的押注,但这个 skill 并不打算自己去做这两类决定——伙伴 skill 各自负责这块领域,每一个都是可选的,且只会被建议一次:

- **[product-manager](../product-manager/)** —— 用于高一个层级的问题:*应该做什么*、按什么顺序做,而不是已经承诺的工作能不能按时交付。它的 roadmap 和 PRD 会作为带来源标签的输入,进入这个 skill 的 `context/charter.md` 和 `plan/`。安装:`npx skills add tronghieu/agent-skills --skill product-manager`
- **[strategy-board](../strategy-board/)** —— 用于超出项目层面的决策:kill/继续、大幅度重新定 scope、portfolio 层面的投资决定——这些不该由一份交付 playbook 自己拍板。董事会给出的建议会作为一条被记录的决策返回,可能还会附带一个变更请求。安装:`npx skills add tronghieu/agent-skills --skill strategy-board`
- **[critical-thinking](../critical-thinking/)** —— 当一个高风险的项目决策建立在一个存在争议的论证之上时使用:vendor 的说辞、一份救援计划背后的逻辑。审查的结论会作为证据被引用进决策日志。安装:`npx skills add tronghieu/agent-skills --skill critical-thinking`

如果某个伙伴 skill 没有安装,这个 skill 只会提一次;如果你不想加装,它会自己继续往下走。永远不是强制项,主动权在你手里。

## 如何触发

试着让你的 AI agent 处理这样的任务:

```text
启动一个新项目——移动端 App 改版,Q3 上线。
```

```text
帮我给这个数据迁移项目建一份风险登记表。
```

```text
deadline 延期了两周——影响有多大?我该怎么跟 sponsor 说?
```

```text
写一份这周给 steering committee 的状态报告。
```

```text
Quản lý dự án này giúp tôi — chuẩn bị báo cáo tiến độ cho sếp tuần này.
```

```text
プロジェクトのリスクレジスターを作成してください。
```

只要工作树里存在 `_project/` 目录,你问任何关于日程、deadline、风险、状态、scope、预算或利益相关者的问题,都会触发这个 skill——你完全不需要说"请扮演项目经理"它才会启动。

## 安装方式

### 1. 使用 CLI（推荐）

```bash
npx skills add tronghieu/agent-skills --skill project-manager
```

### 2. 手动安装（适合不熟悉技术的用户）

1. **下载:** 前往本仓库的 `skills/` 文件夹,下载 `project-manager.zip`。
2. **解压并复制:** 解压 `project-manager.zip`,把 `project-manager` 文件夹复制到以下目录之一:

**针对某个特定项目:**
把 `project-manager` 文件夹复制到项目根目录下的 `.agents/skills/` 或 `.claude/skills/`。

**全局安装（对所有项目生效）:**
* **Mac / Linux:** `~/.agents/skills/` 或 `~/.claude/skills/`
* **Windows:** `%USERPROFILE%\.agents\skills\` 或 `%USERPROFILE%\.claude\skills\`（通常是 `C:\Users\<你的用户名>`）

## 使用示例

> **你:** 写一下这周的状态报告——今天下午要发给 steering committee,说实话 M3 现在情况不太好。
>
> **Agent:** 在动笔之前——工作区显示 M3 上周就因为 vendor API 延期(`R4`)被标记为黄色,而且目前没有任何证据表明这个延期已经解决。如果我写成绿色,那就是一份正在酝酿的"外绿内烂"报告。我该照实标黄并引用 vendor 风险,还是有什么我漏掉的事实——vendor 从上周二之后交付了吗?
>
> **你:** 没有,还是卡着。照实报告吧。
>
> **Agent:** *(起草报告:M3 标黄,引用 `R4` 和最近一次 vendor 更新日期作为证据;预测日期顺延,依据写清楚;另外两个里程碑保持绿色,各自附上证据;报告以坏消息开头,成绩随后)*
>
> 草稿在这——M3 以黄色打头,附上 vendor 延期的原因和修订后的预测日期,依据是 `R4` 和有日期记录的状态历史。我已经把它送去过一遍 Judge 审查:有一个发现——这次的预测延期还没有反映出你 lesson 登记表里的规律(`L2` 之前就标记过 vendor 依赖是反复出现的延期来源),所以我把这个背景加进了风险说明里,而不是把这次当成孤立事件处理。已经作为今天的快照记录进 `status/`,黄色的判定也记进了 `decisions.md`,并附上了"vendor 回应后重新评估"的触发条件。

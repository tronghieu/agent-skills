# Group 1 — Creative expansion

Four techniques for opening a topic up before anything narrows: removing
limits, borrowing structure from elsewhere, flipping the question, and
stripping back to fundamentals. Reach for this group early in a session,
especially when the topic is wide open or has gone stale inside its
obvious answers. Each example below uses one running generic topic — a
community tool-lending library (a place where neighbors borrow tools
instead of buying them) — so you can see the same idea explored several
different ways.

## 1. What-if scenarios

**What it is** — A provocative hypothetical that removes a real-world
limit ("what if it were free?") so the topic gets reconsidered from a
position it can't normally stand in. The limit you remove is the whole
point: budget, time, physics, permission, scale — each removed limit
opens a different door.

**Best for** — Wide-open topics with no fixed direction yet, or a
session that's gone flat inside the obvious answers.

**Facilitate it** — Open with one concrete hypothetical: "What if
[topic] had unlimited budget?" Each round swaps in a new axis — budget,
time, audience, scale, permission. Keep each what-if singular; stacking
two constraints at once ("unlimited budget AND no rules") muddies which
removed limit produced which idea, so the user can't tell what to credit.

**Example round**
- Prompt: "What if the tool library had unlimited budget — no cost
  constraint at all?"
- *(user)*: "We'd buy one of everything, including expensive stuff like
  pressure washers and generators."
- *(AI) builds*:
  - *(AI) Scale up the ambition*: not just tools — a fabrication shop
    with a laser cutter and 3D printers members could book time on.
  - *(AI) Remove the friction entirely*: a van that delivers the tool to
    your door and picks it back up, so "borrowing" costs zero effort.
  - *(AI) Flip who pays*: unlimited budget could also mean paying members
    a stipend to lend out tools they already own.
- Return the floor: "Which of those pulls at you — or is there a
  cheaper version of one of them worth chasing even without the
  unlimited budget?"

**Your build move** — Push the user's answer one step further than they
took it, or apply the same hypothetical to a part of the topic they
didn't address (they answered for tools; what about the building, the
staff, the membership model?).

**Where it goes wrong** — What-ifs that are too timid produce answers
indistinguishable from the status quo ("what if we had a bit more
budget?" gets "we'd buy a few more tools"). If a round comes back flat,
the hypothetical wasn't provocative enough — escalate it hard on the
next round instead of accepting a small idea as the ceiling.

**Bail when** — Three or four hypotheticals in a row produce minor
variations on the same idea; switch to Analogical thinking or SCAMPER to
change the angle of attack entirely.

## 2. Analogical thinking

**What it is** — Borrowing the structure of a well-understood domain and
mapping it onto the topic. Unrelated-looking domains often share a shape
— a lifecycle, a set of roles, a tension — that suggests moves the
topic's own field would never generate on its own.

**Best for** — Topics where the field's own vocabulary and reference
points have gone stale.

**Facilitate it** — Seed one analogy to show the shape of the move:
"[Topic] is a bit like a library of books — you check things out,
they're due back, someone tracks what's overdue. What does that
suggest?" Then ask the user for two or three analogies of their own, and
mine each one in turn: what does the analogy's structure suggest the
topic is currently missing?

**Example round**
- Prompt: "The tool library is a bit like a public library of books —
  due dates, holds, overdue fines. What does that suggest for us?"
- *(user)*: "Maybe a hold system — if someone wants the one ladder we
  have, they can reserve it instead of just showing up and missing out."
- *(AI) builds*:
  - *(AI) Extend the book-library analogy further*: "recommended for
    you" — if you borrowed a stud finder, the system suggests a level
    and a tape measure.
  - *(AI) A second analogy — a gym*: memberships with tiers, a "day
    pass" for non-members, equipment orientation sessions before first
    use of power tools.
  - *(AI) A third analogy — a seed library*: tools that get "returned"
    slightly used or improved (sharpened, oiled) the way seed libraries
    ask growers to save and return seed from what they grow.
- Return the floor: "Want to dig into the hold system more, or follow
  one of the other two analogies somewhere?"

**Your build move** — Extend the user's chosen analogy one layer deeper
rather than introducing a competing analogy of your own — depth in one
analogy usually beats breadth across several.

**Where it goes wrong** — The analogy gets treated as decoration instead
of structure: the user says "it's like Airbnb" and stops there without
mapping any specific mechanic across. If that happens, ask directly:
"what's one specific mechanic from Airbnb — not the vibe, the mechanic —
that would change how this works?"

**Bail when** — The user's analogies keep restating the same one or two
domains (everything maps back to "it's like a library"); that's a sign
the topic needs a different opening move, like What-if or Reversal.

## 3. Reversal / Inversion

**What it is** — Instead of asking how to solve the problem, ask how to
cause the opposite outcome or actively make things worse — then flip
each answer around. Naming a failure mode people avoid saying out loud
is, once flipped, a solution direction in disguise.

**Best for** — Improving something that already exists, or a topic where
direct brainstorming keeps circling the same safe answers.

**Facilitate it** — Open with: "How would we guarantee the tool library
fails, or make it actively worse?" Let worsening moves flow freely — this
half stays destructive, no flipping yet. Once a batch is down, flip each
one in a separate beat: "so the opposite of that is…?" Keep the two
beats distinct; flipping mid-brainstorm cuts the destructive energy
short before it's generated anything useful.

**Example round**
- Prompt: "How would we guarantee the tool library fails within a year?"
- *(user)*: "Never explain how to use the power tools safely, lose track
  of what's been returned, and make the checkout process take twenty
  minutes."
- *(AI) builds* (the flips, as a separate beat):
  - *(AI) Flip "never explain safety"*: a two-minute video required
    before first checkout of any power tool, tied to the specific tool.
  - *(AI) Flip "lose track of returns"*: a simple tag-in/tag-out board at
    the counter, visible to everyone, not buried in a spreadsheet.
  - *(AI) Flip "twenty-minute checkout"*: a phone-based self-checkout for
    repeat members, staffed counter only for first-timers.
- Return the floor: "Any of those worth designing out further, or does
  flipping one of your other worsening moves point somewhere better?"

**Your build move** — Flip one of the user's "make it worse" answers
that they didn't flip themselves — the one they skipped is often the
most revealing, because people tend to avoid naming their own actual
fear.

**Where it goes wrong** — The worsening round turns into a complaint
session about a real, current problem instead of a generative exercise,
and the energy curdles into blame. If it drifts that way, redirect:
"good, that's real — now let's guarantee failure on a clean slate,
nothing that's happened yet."

**Bail when** — The flips start reproducing the existing plan almost
exactly (nothing new is surfacing); that means the destructive half
wasn't pushed far enough, or the topic is ready to converge rather than
diverge further.

## 4. First-principles thinking

**What it is** — Strip the topic down to what's fundamentally true about
it — physical constraints, the actual need underneath it, real costs —
discarding everything inherited from "how it's always been done," then
rebuild options from that bare foundation.

**Best for** — Deadlocks where every idea generated so far is a
variation on the same existing approach.

**Facilitate it** — Ask: "Strip away how tool libraries are usually run
— what's actually, fundamentally true here? What does this have to do,
no matter who builds it?" Once a fundamental is named, the round becomes:
"given just that fundamental, what's one way to meet it that looks
nothing like the standard approach?"

**Example round**
- Prompt: "Strip it down — what's fundamentally true about lending
  tools, no matter how it's organized?"
- *(user)*: "People need a tool for a short window, don't want to own it
  forever, and need to trust it'll work when they get it."
- *(AI) builds*:
  - *(AI) Meet "short window" differently*: skip a central building
    entirely — a rotating tool locker at five neighborhood spots, no
    fixed "library" location at all.
  - *(AI) Meet "don't want to own it forever" differently*: a
    tool-swap network with no central inventory — members list what
    they own and lend peer-to-peer, the "library" is just the directory.
  - *(AI) Meet "trust it'll work" differently*: a simple return-condition
    photo logged at drop-off, so trust comes from a visible track record
    per tool, not a central inspector.
- Return the floor: "Any of those worth chasing even though they don't
  look like a 'library' at all?"

**Your build move** — Point out an inherited assumption still hiding
inside the user's "fundamental" (e.g. "don't want to own it forever"
still assumes ownership is the default state to escape from) — the
first pass at bedrock is usually one layer short of true bedrock.

**Where it goes wrong** — The "fundamentals" the user names turn out to
be assumptions in disguise ("it has to have a physical location" isn't
fundamental, it's inherited). Test each fundamental by asking "is that
actually true, or just how it's always been done?" before building on it.

**Bail when** — The rebuilt ideas keep converging back to something
close to the original approach; that's a sign the true fundamentals
were already close to what exists, and the session is better served by
SCAMPER (improve what's there) than more first-principles digging.

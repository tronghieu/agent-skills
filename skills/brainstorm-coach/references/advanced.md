# Group 5 — Advanced

Seven techniques for when the session needs something sharper than
open-ended ideation: forcing a bridge between unrelated concepts,
reversing a load-bearing assumption, stepping into someone else's shoes,
relocating the problem to a different era, imposing a hard constraint,
sustaining one metaphor as an engine, or generating nothing but
questions. Reach for this group to challenge a topic's frame, to see it
through eyes other than the user's own, or to work against constraints
that feel either suffocating or entirely absent. The examples below use
one running generic topic — a community tool-lending library (a place
where neighbors borrow tools instead of buying them) — so the same idea
can be seen through several techniques.

## 14. Forced relationships

**What it is** — Taking two genuinely unrelated concepts and forcing a
bridge between them, on the premise that the bridge itself — not either
concept alone — is where the new idea lives.

**Best for** — Similar territory to Random stimulation, but for a
sustained pairing held across a couple of rounds rather than a
single-word jolt.

**Facilitate it** — Name an unrelated second concept and ask for the
bridge: "Here's an unrelated thing: a bicycle. What's the bridge between
that and the tool library — what do they have in common, or what would
happen if you combined them?" Hold the same pairing across a couple of
rounds before switching to a fresh second concept.

**Example round**
- Prompt: "Here's an unrelated thing: a bicycle. What's the bridge
  between a bicycle and the tool library?"
- *(user)*: "A bicycle gets you places under your own power — maybe the
  library should be about building capability, not just lending stuff.
  Like, you leave knowing how to do something, not just holding a tool."
- *(AI) builds*:
  - *(AI) A second bridge from the same pairing*: bicycles need regular
    small maintenance (oiling a chain) — maybe tools need the same, and
    the library becomes the place that also teaches basic upkeep for
    tools people already own, not just lending new ones.
  - *(AI) A third bridge*: bikes are often shared via docking stations
    scattered around a city — the tool library could adopt that
    physical pattern, small unstaffed docks instead of one central hub.
- Return the floor: "Does 'you leave knowing how to do something' feel
  like the strongest thread, or one of these other two?"

**Your build move** — Find a second bridge point between the same two
concepts, different from the one the user found — a forced relationship
usually has more than one seam worth pulling on.

**Where it goes wrong** — The user's bridge stays at the level of vague
vibe ("a bicycle is freeing, and so is a tool library") without landing
on anything concrete enough to act on. Push for a mechanic, not a mood:
"okay, but what's one specific thing bicycles actually do that this
could copy?"

**Bail when** — Two pairings in a row produce only vibe-level
connections with no concrete mechanic; switch to Random stimulation
(often an easier on-ramp) or Analogical thinking (which invites a
fuller structural map instead of a single bridge).

## 15. Assumption reversal

**What it is** — Surfacing the assumptions everyone treats as fixed
about the topic, then deliberately reversing each one and building an
idea from inside the reversal — the assumption nobody questions is
usually the one most worth questioning.

**Best for** — Challenging the frame of the topic itself, not just
generating ideas within it; pairs well with Question storming for a
frame-challenging session.

**Facilitate it** — Ask for the assumptions first: "What are we all
assuming is true about the tool library that we've never actually
questioned?" Once assumptions are listed, reverse one per round:
"reverse that — what if the opposite were true instead? What idea does
that open up?"

**Example round**
- Prompt: "What are we assuming about the tool library that we've never
  questioned?"
- *(user)*: "That it needs a physical location. That people want to own
  fewer tools, not more. That borrowing is cheaper than buying."
- *(AI) builds* (reversing one the user didn't reverse themselves)*:
  - *(AI) Reverse "borrowing is cheaper than buying"*: what if it's not
    about cost at all — some members might pay a premium specifically
    for the convenience and lack of storage burden, which reframes
    pricing away from "cheap alternative" toward "premium convenience."
  - *(AI) Reverse "people want to own fewer tools"*: what if some
    members actually want to own more, but need to try before they buy
    — the library becomes a tool showroom that drives eventual
    ownership, with a purchase discount for frequent borrowers.
- Return the floor: "Does either of those reversed assumptions change
  how you'd want to position this?"

**Your build move** — Take an assumption the user reversed but didn't
build on ("what if we didn't need a fixed location") and propose the
concrete idea that reversal makes possible, rather than leaving the
reversal abstract.

**Where it goes wrong** — The "assumptions" the user names turn out to
be things nobody actually believes (straw-man assumptions chosen because
they're easy to reverse), which produces reversals that feel hollow. Ask
"who actually holds this assumption, and would they push back if you
said it out loud?" to test whether it's load-bearing.

**Bail when** — The reversals keep producing ideas that are interesting
but clearly unworkable rather than provocative-but-buildable; that's a
sign the assumptions chosen weren't the real load-bearing ones — go back
and ask for the assumptions again, pushing harder on "what would
genuinely surprise a regular user of this thing?"

## 16. Role playing

**What it is** — Generating ideas as a specific stakeholder or persona
would — a competitor, a skeptical customer, a ten-year-old, a city
inspector — rather than from the user's own vantage point.

**Best for** — Seeing the topic through eyes it hasn't been seen
through; pairs well with Six Thinking Hats for a multi-perspective
session.

**Facilitate it** — Name the persona and ask the user to answer as that
persona: "Step into the shoes of someone who's never borrowed a tool in
their life and is a little intimidated by the idea — what would they
want from the tool library, or what would make them turn around and
leave?" Hold the same persona for a full round or two before switching;
hopping personas mid-round dilutes the perspective each one offers.

**Example round**
- Prompt: "As someone intimidated by tools and unsure what half of them
  even do — what would make you turn around and leave the tool library?"
- *(user)*: "If everyone there seemed to already know what they were
  doing and I felt dumb asking what a thing was for."
- *(AI) builds*, staying inside the same persona:
  - *(AI, in persona)*: "Also — if the tools were labeled with brand
    names and model numbers instead of what they're actually for, I
    wouldn't even know what to search for."
  - *(AI, in persona)*: "And if I had to commit to a membership before
    I even knew if this was for me — I'd want to just try borrowing one
    simple thing first, no commitment."
- Return the floor: "What would make this persona feel welcome instead —
  still speaking as them, or want to switch back to your own voice?"

**Your build move** — Stay inside the same persona the user just voiced
and add what that persona would say next — never break character to
add a neutral idea, since the value here is the constraint of staying
in the persona's voice.

**Where it goes wrong** — The persona collapses into a caricature (the
"skeptical customer" just says "no" to everything) instead of a
believable person with specific, situational concerns. If answers get
generic, ground the persona with one more concrete detail: "what's this
person's actual day like — what were they doing right before they
walked in?"

**Bail when** — The persona's answers start sounding identical to the
user's own default perspective (the "skeptic" agrees with everything);
that means the persona isn't different enough from the user to be
useful — pick a more sharply contrasting persona or switch to Six
Thinking Hats for a more structured multi-perspective pass.

## 17. Time shifting

**What it is** — Relocating the topic to a different era — decades in
the past or the future — where today's tools, norms, or technology don't
apply, forcing solutions built from whatever was or will be actually
available.

**Best for** — Topics too anchored in "how we'd do it with today's
tools" — useful when the obvious answer is always some version of "just
build an app for that."

**Facilitate it** — Name the era: "Solve this as if it were 1985 — no
smartphones, no internet, no way to check inventory remotely. What would
you actually do?" or "…as if it were 2040 — what's taken for granted by
then that changes everything?" Hold one era per round; switching eras
every answer breaks the constraint that makes this technique work.

**Example round**
- Prompt: "Solve the tool library as if it were 1985 — no smartphones,
  no internet, nothing digital at all."
- *(user)*: "A physical logbook at the counter. A paper card for each
  tool that gets stamped with a return date and signed out to whoever
  has it."
- *(AI) builds*, staying inside 1985:
  - *(AI) Push further into what 1985 really offered*: a community
    bulletin board out front where people post what tool they need for
    the week — a low-tech matching system that doesn't need a database
    at all, just paper and foot traffic.
  - *(AI) Another 1985-native idea*: a phone tree — a volunteer calls
    around the neighborhood when a rare tool comes back in, the way
    school snow-day trees worked.
- Return the floor: "Does the bulletin-board matching idea have a
  version worth bringing back into today, even with all our modern
  tools available?"

**Your build move** — Push further into what the chosen era actually
had — if a "1985" answer still quietly assumes a phone app, point out
what 1985 really offered instead (paper, phone trees, bulletin boards,
face-to-face) and build from that.

**Where it goes wrong** — The user's era-shifted answer sneaks in
today's technology anyway ("in 1985 they'd have used a spreadsheet" —
they wouldn't have). Catch this gently and redirect to what genuinely
existed in that era.

**Bail when** — The era-shifted ideas turn out to be interesting
history but don't translate into anything usable today; that's fine as
a one-round detour, but if a second era also produces nothing
transferable, move to a technique aimed more directly at the present
constraint, like Resource constraints.

## 18. Resource constraints

**What it is** — Imposing a severe, specific limit — money, time,
people, tools — and forcing the idea to survive inside it, on the
premise that real constraints produce sharper, more inventive ideas than
"anything is possible."

**Best for** — Sessions where constraints feel suffocating (reframe the
real constraint as a game) or, just as often, where constraints feel
entirely absent and the topic needs an artificial one to get specific.

**Facilitate it** — State one concrete limit: "What if you had $200 and
one weekend to get the tool library idea into the world — what would you
actually do?" Vary the limit each round — money, then time, then
headcount, then physical space — one axis at a time so the constraint
stays sharp instead of vague.

**Example round**
- Prompt: "What if you had $200 and one weekend — what would you
  actually launch?"
- *(user)*: "I'd buy five of the most commonly needed tools — a drill,
  a ladder, a stud finder, a wet-dry vac, a pressure washer nozzle — put
  them in my garage, and post in the neighborhood group that people can
  borrow them, tracked on a shared spreadsheet."
- *(AI) builds*:
  - *(AI) Cut one more corner*: skip the spreadsheet entirely for
    weekend one — a paper sign-out sheet taped to the garage wall does
    the same job with zero setup time.
  - *(AI) Push the constraint further*: with only $200, don't buy tools
    at all — post asking five neighbors to each commit one tool they
    already own for the weekend, turning the $200 into pizza for a
    launch potluck instead.
- Return the floor: "Does the zero-purchase, borrow-from-neighbors
  version feel more launchable than the $200-of-tools version?"

**Your build move** — Take the user's constrained idea and find one
more corner to cut that they didn't — the tightest version of a
resource-constraint round often comes from the second pass, not the
first.

**Where it goes wrong** — The stated constraint gets quietly ignored
partway through ("$200 and one weekend" somehow includes hiring a
contractor); if the constraint slips, restate it plainly and ask the
user to re-answer inside it.

**Bail when** — The constrained answer is just a smaller version of
whatever was already planned, with nothing genuinely different about
its shape; try a sharper or stranger constraint before abandoning the
technique, and only move on if two different constraints both produce
the same shrunk-down idea.

## 19. Metaphor mapping

**What it is** — Choosing one extended metaphor for the topic and mining
it as an idea engine — every feature of the metaphor becomes a prompt
for a feature of the topic, sustained across the whole round rather than
used once and dropped.

**Best for** — Topics that resist literal thinking, or when a session
needs a shared, memorable frame to organize a lot of ideas at once.

**Facilitate it** — Propose one extended metaphor and hold it: "Let's
say the tool library is a garden. What needs weeding? What needs
watering? What's already in bloom?" Keep pulling new features out of the
same metaphor each round rather than switching metaphors — the depth
comes from staying inside one rather than sampling several shallowly.

**Example round**
- Prompt: "Let's say the tool library is a garden. What needs weeding?"
- *(user)*: "The tools nobody ever borrows — they're just taking up
  shelf space and should probably go to a different library or get sold."
- *(AI) builds*, staying inside the garden metaphor:
  - *(AI) What needs watering*: the volunteer relationships — the
    people who show up to teach and staff the counter need regular
    appreciation or they'll dry up and stop showing.
  - *(AI) What's already in bloom*: the informal tool-tip-sharing that
    happens at the counter when people chat while checking out — worth
    naming and nurturing rather than leaving accidental.
  - *(AI) What needs a trellis (structure to grow well)*: the classes —
    right now ad hoc, they'd grow better with a simple recurring
    schedule to climb.
- Return the floor: "Any of those garden-features worth turning into an
  actual plan, or want to keep pulling more out of the metaphor first?"

**Your build move** — Reach for a feature of the metaphor the user
hasn't used yet (if they've covered weeding and watering, raise pruning,
or what's planted too close together) and translate that specific
feature into a topic idea.

**Where it goes wrong** — The metaphor gets abandoned after one round in
favor of a fresh one, which loses the compounding depth that makes this
technique work. If the user's next answer reaches for a totally
different metaphor, gently pull back: "let's stay in the garden a
little longer — what else does a garden need?"

**Bail when** — The chosen metaphor runs dry — two rounds produce
strained, reaching connections rather than natural ones; that's a sign
the metaphor was a weak match for this topic, not that metaphor mapping
itself failed — try a different metaphor once, and if that also strains,
move to a different technique.

## 20. Question storming

**What it is** — Generating only questions about the topic, no answers
allowed, for the whole round — then, at the end, picking the questions
that feel most generative to actually pursue. Suspending the pull toward
answering surfaces questions a normal brainstorm skips past on the way
to a solution.

**Best for** — Challenging the frame of the topic, and as a strong
converging move late in a progressive flow, right before synthesis.

**Facilitate it** — Set the rule explicitly up front: "This round is
questions only about the tool library — no answering, not even a hint
of one, just raise the question and move to the next." Let the user list
several before you add any. At the end of the round, ask the user which
two or three questions feel like they'd open the most doors.

**Example round**
- Prompt: "Questions only about the tool library — no answering."
- *(user)*: "Who's actually liable if someone gets hurt? Does this need
  to be a nonprofit? What happens when a tool breaks — whose problem is
  it? Would the city ever fund something like this?"
- *(AI) builds* (also questions, not answers)*:
  - *(AI)*: "What does 'success' even look like a year in — number of
    members, tools borrowed, or something else entirely?"
  - *(AI)*: "Why does it have to be one central library at all — what
    if the question is really 'how do neighbors share what they own'
    and a library is just one answer to that?"
  - *(AI)*: "What's the actual competitor here — is it hardware store
    rentals, or is it just people not fixing things at all?"
- Return the floor: "Which two or three of these feel like they'd open
  the most doors if we chased them?"

**Your build move** — Add questions the user's list doesn't cover yet —
especially ones that question the topic's framing itself ("why does it
have to be X at all?") rather than questions still inside the existing
frame.

**Where it goes wrong** — An answer sneaks in disguised as a question
("could we just charge a small deposit to cover breakage?" is really a
proposal). Catch these gently — "that's actually an answer wearing a
question mark, let's save it" — and keep the round genuinely
answer-free.

**Bail when** — The questions start repeating the same concern in
different phrasing rather than opening new territory; that's the signal
to stop and move to selecting the most generative ones, which is often
the natural bridge straight into convergence.
